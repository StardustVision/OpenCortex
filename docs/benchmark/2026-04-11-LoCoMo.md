Two facts explain most of the gap:

1. **Non-adversarial zero-retrieval is 42.2%**: 650 / 1540 Cat 1-4 questions returned no memory items.
2. **Latency is above the planning timeout**: p50 is 18.4s while `ContextManager` gives recall planning only 10s before degrading to a fallback plan that disables recall in `auto` mode.

- Retrieval hit rate on Cat 1-4: **57.8%**

- Hit vs miss J-Score: **0.565 vs 0.185**

So this run is not primarily an answer-model collapse. It is a **recall-path collapse**, followed by a thinner-than-needed answer context.

## 8. Code-Level Attribution

### A1. Timeout can degrade `auto` mode into "do not recall"

This is the most important code-path finding.

Flow:

```text
context_recall
  -> ContextManager._prepare()
  -> plan_recall() wrapped in asyncio.wait_for(..., timeout=10.0)
  -> if timeout/error, keep fallback RecallPlan
  -> fallback RecallPlan in auto mode sets should_recall = False
  -> retrieval skipped
```

Historical code at run time:

- `src/opencortex/context/manager.py`
- `src/opencortex/orchestrator.py`
- legacy router/planner path deleted in the 2026-04-13 refactor

Why this matters here:

- measured recall latency p50 is **18.4s**
- planning timeout is **10.0s**
- fallback plan for `auto` mode disables recall entirely

This is an inference from code + observed latency, but it is a strong one: timeout-induced no-recall is very likely a first-order contributor to the 42.2% zero-retrieval rate.

### A2. Historical run path: every `context_recall` request still went through LLM intent planning

At run time, the old router path still ran LLM classification whenever
`session_context` was present and an LLM callable existed:

- historical Phase-1 router path at run time, since replaced by
  `src/opencortex/intent/probe.py`

`ContextManager._prepare()` always passes a `session_context` into `plan_recall()`:

- `src/opencortex/context/manager.py:307-323`

So the current production recall path pays LLM planning cost even for straightforward benchmark questions. That cost sits directly on the critical path before retrieval.

### A3. Temporal signal is stripped out of the embedded text

LoCoMo observations are ingested as:

- `abstract = "[date] speaker: text"`
- `embed_text = "speaker: text"`

Relevant code:

- `benchmarks/adapters/conversation.py:208-220`

That means the date is visible in the stored abstract, but **not in the vector text used for embedding**. This is a concrete code-level reason Cat 2 temporal questions underperform.

### A4. Observations are stored independently, without a relation layer

The benchmark ingests each observation as a separate record:

- `benchmarks/adapters/conversation.py:203-220`

There is no explicit entity graph, event graph, or multi-observation join structure in this benchmark path. Cat 4 open-domain / multi-fact questions therefore rely on a single query recovering several independent facts from a flat observation pool.

### A5. The benchmark pins answer context to `l0`

`context_recall` is called with `detail_level="l0"`:

- `benchmarks/adapters/conversation.py:503-509`

And `DetailLevel.L0` is abstract-only:

- `src/opencortex/retrieve/types.py:31-36`

This is the most plausible explanation for the 200 cases where exact evidence overlapped but the final judged answer still failed. The retriever found something relevant, but the answer model saw only the thinnest representation.

### A6. Search failures can silently degrade into empty results

`ContextManager` catches exceptions in memory and knowledge search and returns empty lists:

- `src/opencortex/context/manager.py:351-380`
- `src/opencortex/context/manager.py:382-397`

This is probably not the main cause of the benchmark gap, but it makes debugging harder and can convert transient infra failures into apparent "retrieval miss" events.

### A7. IntentRouter cache reset is real, but secondary for this benchmark

`Orchestrator.plan_recall()` constructs a new `IntentRouter` for each call:

- `src/opencortex/orchestrator.py:1833-1839`

That destroys the in-router LRU cache. However, LoCoMo benchmark questions are mostly unique, so cache loss is a real defect but probably **not** the dominant cause of this specific run's latency.

## 9. Optimization Roadmap

| Priority | Action | Why it matters | Expected effect |
|----------|--------|----------------|-----------------|
| **P0** | Make planner timeout fail open for recall | Current fallback in `auto` mode disables recall entirely | Directly cuts zero-retrieval caused by timeout |
| **P0** | Add a no-LLM fast path for factual / temporal benchmark-style queries | LLM planning is on the critical path for every request | Large latency reduction; fewer timeout cascades |
| **P1** | Include session date and normalized time tokens in `embed_text` | Temporal signal is currently absent from embeddings | Improves Cat 2 temporal retrieval |
| **P1** | Use adaptive detail level: promote exact-hit answer generation from `l0` to `l1` | 200 exact-hit cases still fail at answer time | Better answer synthesis without huge token growth |
| **P1** | Tune hybrid retrieval for date/name-heavy queries | BM25/sparse already exists; current path needs stronger lexical help on exact queries | Better Cat 1 and Cat 2 precision |
| **P1** | Decompose Cat 4 questions into multiple typed subqueries | Flat single-query retrieval under-serves multi-fact prompts | Better Cat 4 evidence coverage |
| **P2** | Emit explicit recall-skip reasons in benchmark output | Today timeout/no-recall is inferred indirectly | Faster diagnosis, less guesswork |
| **P2** | Stop swallowing retrieval exceptions into silent empty lists | Empty results currently conflate miss vs infra failure | Cleaner observability and safer ops |

### Recommended implementation order

1. **Planner timeout policy**
2. **No-LLM fast path**
3. **Temporal embedding fix**
4. **Adaptive `l1` answer context**
5. **Hybrid tuning + multi-query decomposition**

If only one thing is fixed first, it should be **"timeout no longer means should_recall=False"**.

## 10. Bottom Line

OpenCortex's main advantage on this LoCoMo run is still **efficiency**, not benchmark quality:

- 98.7% token reduction is excellent
- but J-Score is **far below** both the in-run baseline and every repo-tracked competitor

The strongest evidence points to a production-path problem:

- **planning is expensive**
- **timeout handling is too aggressive**
- **temporal information is missing from the embedded text**
- **answer context is too thin when retrieval succeeds**

This is fixable. The benchmark does **not** show a fundamentally bad memory system. It shows a recall pipeline that currently drops too many requests before retrieval and then under-feeds the answer stage on the requests that survive.
