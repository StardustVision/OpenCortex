## 2. 指标详解

### 2.1 J-Score（主指标）

**定义：** 基于 LLM 的二值判定评分（对齐 Mem0 评测标准）。将问题、标准答案和模型预测一起交给 LLM 评判，输出 1（正确）或 0（错误）。

**计算方式：**
```
对每个 QA 对:
  评判提示 = f"问题: {q}\n标准答案: {gt}\n模型预测: {pred}\n预测是否正确？(1/0)"
  jscore = LLM(评判提示) → 0 或 1
总分 = 所有 jscore 的均值
```

**解读：**
- 1.0 = 所有回答被 LLM 判定为正确
- 0.8 = 80% 的回答被判定为正确
- 优于 F1 的原因：J-Score 捕捉 **语义正确性**，而非词汇重叠。例如 "北京" 和 "中国首都" 语义相同但 F1=0

**适用场景：** 所有数据集的主要评估指标

### 2.2 F1 分数（辅助指标）

**定义：** 预测文本与标准答案之间的 Token 级 F1 重叠度。计算前对文本做归一化处理（小写化、去标点、去冠词）。

**公式：**
```
精确率 = |预测词 ∩ 标准词| / |预测词|
召回率 = |预测词 ∩ 标准词| / |标准词|
F1 = 2 × 精确率 × 召回率 / (精确率 + 召回率)
```

**解读：**
- 衡量的是词汇重叠，**不是**语义正确性
- 语义正确但措辞不同时 F1 偏低（假阴性）
- 预测包含正确词但夹杂大量无关词时 F1 偏高（假阳性）
- PersonaMem 的 F1 普遍很低（0.15 左右），是因为标准答案是长文本，词汇重叠天然较少

### 2.3 精确匹配（EM）

**定义：** 二值指标（0/1）— 归一化后的预测文本是否与标准答案完全一致？

**报告范围：** 仅 HotPotQA（多跳问答标准指标）

### 2.4 证据事实 F1（SP F1）

**定义：** 检索到的文档标题集合与标注的证据文档标题集合之间的集合级 F1。

**公式：**
```
SP_precision = |检索到的标题 ∩ 标注标题| / |检索到的标题|
SP_recall    = |检索到的标题 ∩ 标注标题| / |标注标题|
SP_F1        = 2 × SP_precision × SP_recall / (SP_precision + SP_recall)
```

**报告范围：** 仅 HotPotQA。衡量系统是否找到了正确的证据文档。

### 2.5 Token 压缩率

**定义：** 发送给 LLM 的提示 Token 数量减少的百分比。

**公式：**
```
压缩率 = (1 - OC 平均 Token 数 / 基线平均 Token 数) × 100%
```

**解读：**
- **基线** = 全量上下文（所有可用文档/消息），截断至预算上限（32k tokens）
- **OpenCortex** = 仅检索出的相关上下文
- 越高越好 — 用更少的 Token 传递关键信息
- 97% 压缩率意味着 OC 只用 3% 的 Token 就完成了回答

**为什么重要：** Token 压缩直接降低 LLM API 成本。97% 压缩率意味着 **LLM 调用成本降低约 30 倍**。

### 2.6 检索延迟

**定义：** OpenCortex `search()` API 从请求到响应的端到端耗时，在客户端测量。

**百分位数含义：**
- **p50（中位数）：** 典型用户体验 — 一半的请求快于此值
- **p95（尾部延迟）：** 绝大多数用户的最差体验
- **p99（极端尾部）：** 压力/边缘场景

**延迟组成：**
```
JWT 认证 → HTTP 传输 → 意图路由 → 文本嵌入 → 向量搜索 → 重排序 → 结果组装
  ~5ms       ~10ms      2-5s(LLM)   ~200ms      ~50ms       ~500ms    ~200ms
```

**主要瓶颈：** LLM 意图分析（2-5s），在有会话上下文时触发。关闭意图分析（仅关键词模式）可将 p50 降至 1-2s。

### 2.7 检索质量（Recall@k、Precision@k、MRR）

**定义：** 衡量系统是否检索到了正确的源文档/段落。

| 指标 | 含义 |
|------|------|
| **Recall@k** | top-k 结果中包含的相关文档占全部相关文档的比例 |
| **Precision@k** | top-k 结果中相关文档占 top-k 总数的比例 |
| **MRR** | 第一个相关结果的排名倒数的均值 |
| **Hit Rate@k** | top-k 中是否至少包含一个相关文档（0/1） |

**本次结果说明：** 所有数据集的检索质量指标均为 0.0。这是一个已知问题 — 评测时的 `expected_uris`（标注的正确 URI）与实际导入时生成的 URI 不匹配（不同的 tenant ID、URI 格式）。检索质量应通过下游 QA 指标（J-Score、F1）间接评估。

- Token 压缩率低（9.2%），因为 HotPotQA 每题上下文本身就很小（~2k tokens）
- SP F1 = 0.446 表明约一半的正确证据文档被成功检索

### 3.4 QASPER — 学术论文问答

**来源：** QASPER dev 集（Allen AI）— 针对 NLP 研究论文的问答
**任务：** 回答关于特定论文内容的问题（抽取式、生成式、是/否、无法回答）
**QA 数量：** 100 | **导入文档：** 34 篇论文

| 指标 | 基线 | OpenCortex | 差值 |
|------|------|------------|------|
| J-Score | 0.800 | 0.150 | **-0.650** |
| F1 | 0.369 | 0.093 | -0.276 |

**Token 统计：**
- 基线平均：6,863 tokens | OC 平均：781 tokens | 压缩率：**88.6%**

**延迟：** p50: 8,852ms | p95: 13,324ms | p99: 14,556ms | 均值: 8,543ms

**分析：**
- **表现最差的数据集** — J-Score 从 0.80 骤降至 0.15
- **根本原因：** 文档分块将论文切割为孤立的段落。当问题需要某个特定段落的信息时，基于嵌入的检索往往无法在 Qdrant 中数百个分块里找到正确的那个
- 基线直接提供整篇论文（~7k tokens）作为上下文，LLM 拥有完整信息。OC 只提供 781 tokens 的检索分块，经常来自错误的章节
- **核心改进方向：**
  - **分块检索精度：** 当前基于标题的分块策略可能将关键段落拆分到不同分块中
  - **父子层级遍历：** 当某个分块命中时，层级检索应提升同一文档内其他分块的权重
  - **文档范围限定搜索：** QASPER 的问题总是针对特定论文的 — 检索应限定在该论文的分块范围内

## 4. 跨数据集洞察

### 4.1 Token 压缩率与准确率的权衡

| 数据集 | Token 压缩率 | J-Score 差值 | 权衡比（每 10% 压缩的 J-Score 损失）|
|--------|-------------|-------------|--------------------------------------|
| HotPotQA | 9.2% | -0.06 | 0.065 |
| LoCoMo | 97.7% | -0.25 | 0.026 |
| PersonaMem | 97.6% | -0.11 | **0.011**（最优） |
| QASPER | 88.6% | -0.65 | 0.073（最差） |

**PersonaMem 提供了最佳的压缩-精度权衡：** 97.6% 压缩率仅损失 0.11 J-Score。
**QASPER 的权衡最差：** 88.6% 压缩率却损失了 0.65 J-Score。

**延迟分解（估算）：**

**优化建议：** 关闭 LLM 意图分析（使用关键词模式）可将 p50 从 ~8s 降至 ~1-2s。

## 5. 性能优化影响

本次测试前合并了 6 项热路径性能优化：

| 优化项 | 内容 | 预期效果 | 实际效果 |
|--------|------|----------|----------|
| 远程嵌入缓存 | volcengine/openai 嵌入器包装 LRU 缓存 | 减少重复 API 调用 | 本次使用本地嵌入，未触发 |
| 访问统计批量化 | 1 次 filter + N 次并行 update | 减少搜索后写放大 | 已包含在延迟中 |
| frontier 饥饿批量查询 | N 个小查询合并为 1 个批量查询 | 减少检索往返次数 | 已包含在延迟中 |
| 冷启动异步化 | 迁移/重嵌入延迟到后台执行 | 加速服务器启动 | 服务器 ~30s 就绪 |
| 结果组装批量预取 | relations 和 abstracts 批量读取 | 减少 FS 扇出 | 已包含在延迟中 |
| 批量导入并发化 | asyncio.Semaphore(8) 有界并发 | 加速批量导入 | 未在此测试中验证 |

**延迟对比（与上次基准测试，优化前）：**
- 上次（优化前）：p50 = 6,970ms | p95 = 9,834ms
- 本次（优化后）：p50 = 7,613ms | p95 = 13,834ms（LoCoMo，最接近的对比）

延迟未在本次测试中显著改善，原因：
1. 服务器包含之前测试的遗留数据（Qdrant 集合更大）
2. LLM 意图分析占据主要延迟（不在本次优化范围内）
3. 优化针对并发场景 — 单用户串行评测无法体现并行化收益

## 6. 改进建议

### P0 — 紧急

1. **修复 QASPER 文档检索：** 实现文档范围限定搜索（按源文档过滤），改进分块边界检测策略
2. **增加对话时序索引：** 索引 `accessed_at`/`created_at` 字段，对时序查询加权近期记忆

### P1 — 重要

3. **降低 LLM 意图分析延迟：** 对相似查询缓存意图结果；考虑使用本地意图分类器替代 LLM
4. **改善 LoCoMo 单跳召回：** 对事实类查询（人名、日期、具体事件）增加关键词/BM25 加权
5. **提高文档模式重排候选数：** 当前默认值可能过于激进地过滤掉了正确的分块

### P2 — 可选

6. **优化文档导入速度：** 文档模式每篇论文 2-3 分钟 — 考虑异步分块处理
7. **修复检索指标对齐：** 使 expected_uri 匹配正确，让 Recall@k 指标可用
8. **运行完整 PersonaMem（2,061 QA）：** 当前 100 QA 样本可能不够代表性
