# OpenCortex 记忆系统增强方案（详细版）

> 日期：2026-03-02  
> 目的：回答“为什么要做 1-5 项增强，以及为什么按该顺序做”

## 1. 结论摘要

你提出的 5 项都成立，且彼此互补：

1. 词法后备检索（无向量时）
2. 访问驱动遗忘（access_count/accessed_at）
3. Canonical Session + 会话压缩
4. 图查询增强（基于 relations）
5. 事件 JSONL 镜像（search/store/feedback）

建议执行顺序：

1. 词法后备检索
2. 访问驱动遗忘
3. 事件 JSONL 镜像
4. Canonical Session + 会话压缩
5. 图查询增强

这个顺序的核心原则是：**先修“召回底座”和“记忆新鲜度”，再补“可观测性”，最后做“结构升级”**。  
即先拿确定性收益，再上改造面更大的能力。

## 2. 当前系统现状（为什么需要增强）

### 2.1 无向量路径缺少语义检索

当前 `QdrantStorageAdapter.search()` 在 `query_vector` 不存在时主要走纯 filter/scroll：

- `src/opencortex/storage/qdrant/adapter.py:377`

这意味着在 `embedding_provider=none` 或向量不可用场景下，系统更像“条件过滤器”，对自然语言 query 的理解能力不足，容易漏召。

### 2.2 遗忘机制主要依赖 reward，不区分“陈旧度”

当前 `apply_decay()` 的衰减对象主要是 `reward_score`：

- `src/opencortex/storage/qdrant/adapter.py:737`

但缺少显式 `accessed_at/access_count` 驱动的“长期未访问才衰减”机制。这样会导致：

- 被动低反馈但近期高频使用的记忆可能被误伤
- 历史脏记忆的下沉速度不够可控

### 2.3 会话压缩还停留在提取阶段截断

当前 `MemoryExtractor` 是“提取时只看最后 50 条消息”的策略：

- `src/opencortex/session/extractor.py:66`

这是必要的 token 保护，但不是完整的会话生命周期管理（没有 canonical session 级别的长期压缩与结构化摘要）。

### 2.4 已有 relations，但缺“图检索层”

`CortexFS` 已支持 relations 读写：

- `src/opencortex/storage/cortex_fs.py:1073`

`HierarchicalRetriever` 也会返回 related abstracts，但目前仍主要是“检索后展示关系”，不是“图传播式检索”。

### 2.5 可观测性不足，难以归因“哪里变好”

当前有 transcript 读取能力（插件侧）：

- `plugins/opencortex-memory/lib/transcript.mjs:1`

但缺统一的 retrieval feedback 事件镜像，无法高质量回答：

- 是 Router 变好，还是 Retriever 变好？
- 是召回变好，还是排序变好？
- 哪类 query 提升最大？

## 3. 为什么 1-5 都有道理（逐项论证）

## 3.1 词法后备检索（优先级 P0）

### 为什么做

- 这是召回系统的“兜底安全网”
- 无向量时不应直接退化为纯过滤；至少应有关键词/词项匹配能力

### 预期收益

- 低成本提升 Recall（尤其是专有名词、路径、错误码）
- 在 embedding 异常或禁用时保持可用性

### 预期指标变化

- Recall@5 上升（首要）
- 空结果率下降
- “硬关键词问题”（如文件名、错误码）命中率上升

### 风险

- 噪声结果增加（词法匹配天生有误召）

### 风险控制

- 词法召回只做候选扩展，不直接替代主排序
- 与现有向量分数融合（RRF 或加权）

---

## 3.2 访问驱动遗忘（优先级 P0）

### 为什么做

- “是否有反馈”不是“是否有价值”的充分条件
- 高频被用到的记忆，应该天然更抗衰减

### 预期收益

- 热门有效记忆保鲜
- 冷门过期记忆自然下沉
- 长期运行后检索稳定性更好

### 预期指标变化

- Top1 稳定性提升
- 错误记忆回流率下降
- 长周期回归中 MRR 更稳

### 风险

- 回写访问统计增加写放大

### 风险控制

- 批量异步回写（采样或聚合）
- 仅对 Top-K 命中做访问计数

---

## 3.3 Canonical Session + 会话压缩（优先级 P1）

### 为什么做

- 当前“提取时截断”是局部优化，不是会话治理
- 多渠道或长会话场景需要统一语义主线

### 预期收益

- 长会话 token 成本可控
- 历史上下文连续性更强
- 关键决策不会被窗口截断丢失

### 预期指标变化

- tokens_with_memory / tokens_without_memory 比值进一步优化
- 长任务平均轮数下降
- “上下文断裂”类错误下降

### 风险

- 压缩摘要如果质量不稳，会引入信息偏差

### 风险控制

- 保留近窗原文 + 压缩摘要并存
- 关键事件（决策/约束/失败原因）做结构化锚点，不只依赖自由文本总结

---

## 3.4 图查询增强（优先级 P2）

### 为什么做

- 复杂问题往往是“关联命中”，不是“单点命中”
- 仅靠向量相似度，跨节点依赖召回不足

### 预期收益

- 多跳关联问题召回提升
- 解释性更好（可说明路径）

### 预期指标变化

- hard query 的 Recall@5 提升
- 关系链问题的正确率提升

### 风险

- 图扩展容易引入噪声，且增加延迟

### 风险控制

- 先做 1-hop/2-hop 限深
- 关系边类型加权（同类边高于弱关联边）
- 仅对前 N 个候选做图扩展

---

## 3.5 事件 JSONL 镜像（优先级 P0.5 / P1）

### 为什么做

- 没有观测就没有优化闭环
- 你需要“知道是哪里变好了”，而不是只看总分

### 预期收益

- 可以做按意图/难度/类别的归因
- 可以做失败聚类，指导下一轮优化

### 预期指标变化

- 直接业务指标不一定立刻上升，但迭代效率显著上升

### 风险

- 日志隐私与存储膨胀

### 风险控制

- 脱敏（token/hash）
- 采样与轮转策略
- 只记录必要字段

---

## 4. 为什么按这个顺序做

## 4.1 依赖关系

- 词法后备检索与访问驱动遗忘是“底层检索质量”改进，属于前置能力。
- 事件镜像是“观测层”，能在前两项上线后立刻回答“收益来自哪里”。
- Canonical Session 与图检索属于“结构性升级”，改造面大，应该在基础指标稳定后推进。

## 4.2 ROI（收益/成本）排序

- P0：1、2（收益快、影响大、可控）
- P1：5、3（一个先补观测，一个强化长会话）
- P2：4（收益高但复杂度最高）

## 4.3 风险管理

如果先做 3 或 4，出现指标波动时难定位根因。  
先做 1/2/5 可以把诊断能力补齐，再上复杂特性，回滚成本更低。

## 5. 分阶段落地计划（建议）

## Phase A（1-2 周）：召回底座 + 观测闭环

范围：

1. 词法后备检索（无向量路径）
2. 访问统计字段与回写
3. JSONL 事件镜像（search/store/feedback）

验收标准（最低）：

- Recall@5 较 baseline 提升 >= 5%
- 空召回率下降 >= 20%
- 日志可完成 query 级归因（可区分 router/retriever/rerank 失败）

## Phase B（1-2 周）：会话治理

范围：

1. Canonical Session 抽象
2. 会话压缩（窗口 + 摘要）
3. 关键事件锚点（决策/约束/失败）

验收标准（最低）：

- 长会话 token 消耗下降 >= 15%
- 长链任务平均轮数下降 >= 10%

## Phase C（2+ 周）：图检索增强

范围：

1. 1-hop/2-hop 邻接扩展
2. 图传播重排（PPR 或简化版）
3. 关系边权重与限深策略

验收标准（最低）：

- hard query 的 Recall@5 再提升 >= 5%
- P95 延迟回归不超过 10%

## 6. 评估与归因（回答“哪里变好了”）

每次搜索最少记录这些字段：

- query_id, query_text_hash
- intent_type, should_recall
- retrieved_uris_topk
- expected_uris（离线集）
- top1_hit, top5_hit
- latency_ms（router/retriever/rerank 分段）
- tokens_injected

最小归因分类：

1. Router 漏召回（should_recall=false 但应召回）
2. Retriever 漏召回（top5 无真值）
3. 排序问题（top5 有真值但 top1 失配）
4. 记忆陈旧/冲突（命中错误旧记忆）

这样就能回答：

- 指标提升来自召回还是排序？
- 哪类 query 改进最明显？
- 哪个模块仍是瓶颈？

---

## 7. 不建议现在优先做的内容

1. 大规模重构存储后端（高风险低确定性）
2. 一次性上完整图数据库迁移（复杂度过高）
3. 没有观测前就并行推进多项复杂算法（难归因）

---

## 8. 最终建议

你的判断“1-5 都有道理”完全正确。  
执行上建议以“**先收益、可观测、再复杂**”为原则推进：

1. 先把召回和遗忘做好（1,2）
2. 立刻补齐归因能力（5）
3. 再推进长会话治理（3）
4. 最后做图检索跃迁（4）

这条路径最稳，也最容易在每个阶段交付可量化收益。
