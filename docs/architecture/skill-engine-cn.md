# 技能引擎（Skill Engine）

## 为什么它存在

OpenCortex 需要一个位置来承载可复用的操作行为，并把它们作为一等工件独立于原始记忆和文档之外。技能是结构化的流程单元（名称、描述、内容、谱系、质量信号），可以被搜索、审批和共享；而记忆仍然承担通用上下文的角色。这样的分离让技能能拥有自己的生命周期（`candidate → active → deprecated`）、质量门控和可见性规则，而不必把这些职责压到记忆子系统上。

## 核心组件

- **SkillManager**：负责检索、抽取、演化、审批、弃用与提升可见性的编排，并对修改类操作执行可见性与归属校验。
- **SkillStore + SkillStorageAdapter**：在独立的 `skills` 集合上提供 CRUD 与向量检索能力，包括可见性过滤和基于嵌入的搜索。
- **SkillAnalyzer**：借助 LLM 提示词，从记忆聚类中抽取候选技能方向，并基于来源指纹去重。
- **SkillEvolver**：通过 LLM 驱动的演化循环（采集型 `captured`、派生型 `derived`、修订型 `fixed`），把建议转为具体的 `SkillRecord` 候选项。
- **QualityGate**：基于规则的校验组件，可选地叠加 LLM 语义检查，并输出质量分数。
- **SandboxTDD**：可选的、由 LLM 模拟的 RED-GREEN-REFACTOR 评估，用来比较基线行为与使用技能后的行为差异。
- **SkillEventStore + SkillEvaluator**：跟踪技能的选择/应用事件，并将其与会话结果关联，用于更新计数器和奖励分数。
- **HTTP 路由**：`/api/v1/skills` 提供列表、搜索、抽取、审批、可见性提升与演化触发等端点。

## 抽取与演化流程

1. **扫描并聚类**：`SkillAnalyzer` 请求源适配器（基于 Qdrant）扫描记忆并完成聚类。
2. **按指纹去重**：每个聚类都会生成确定性的指纹；若已存在相同指纹的技能，则跳过。
3. **LLM 抽取**：分析器会整理聚类内容和现有技能，要求 LLM 返回候选方向。
4. **演化**：采集型 `captured` 技能使用基于指纹派生的 ID；派生型 `derived` 技能使用 UUID，并引用父技能；修订型 `fixed` 技能则不是原地修改已有技能，而是为其创建一个新的候选版本。
5. **创建候选项**：`SkillEvolver` 构建 `SkillRecord`，状态设为 `CANDIDATE`、可见性设为 `PRIVATE`，并附带谱系元数据。

## 校验与审批流程

1. **Quality Gate（阶段 A，可选）**：只有在配置了 LLM 适配器时，这个门控才存在。存在时，它会运行确定性检查（名称格式、内容长度、步骤结构、描述、类别），并可额外加入语义检查。得分低于阈值（60）的候选项会在持久化前被拒绝。
2. **Sandbox TDD（阶段 B，可选）**：如果启用，会基于生成的场景测试候选项。测试失败会阻止持久化。
3. **持久化**：只有通过上述门控的候选项才会被保存到 `skills` 集合。
4. **人工审批**：`approve` 与 `reject` 都是显式 API。只有归属者能变更状态。获批的候选项会转为 `ACTIVE`；被拒绝或被弃用的技能会转为 `DEPRECATED`。
5. **可见性提升**：可见性与状态是两条独立轴线。归属者可以把 `PRIVATE` 技能提升为 `SHARED`（同时重新生成 URI），而当前代码中的提升操作并不以审批为前置条件。

## 检索与 API 暴露

技能引擎通过两条路径对外暴露：

- **REST API**：`/api/v1/skills` 支持 `list`、`search`、`extract`、`get`、`approve`、`reject`、`deprecate`、`promote`、`fix` 和 `derive`。身份与租户范围来自请求上下文。
- **`MemoryOrchestrator.search()` 路径**：在常规记忆检索中，技能检索会被合并进 `FindResult.skills`。这样可以让技能检索与用户查询保持对齐，同时又保持技能存储独立于记忆记录。

技能使用自己的 URI 表示（`opencortex://{tenant}/shared/skills/...` 或 `opencortex://{tenant}/{user}/skills/...`），并会在检索结果中与记忆、资源上下文一同返回。

## 约束与权衡

- 技能存放在独立的 Qdrant 集合中，并有自身的可见性规则；这简化了访问控制，但也意味着需要额外维护一层存储面。
- 抽取与演化依赖 LLM；如果没有配置 LLM，整条流水线会被禁用，只剩手动管理能力。
- Quality Gate 与 Sandbox TDD 能降低低质量候选项混入系统的概率，但也可能拦下有价值的草稿，并增加延迟。
- 审批是人工且按归属者约束的；当前没有基于使用指标的自动提升。
- 引擎当前依赖记忆聚类而不是执行轨迹做抽取，因此更偏向总结通用模式，也可能遗漏任务级细节。

## 当前状态

技能引擎已经在 `MemoryOrchestrator.init()` 中接线完成：

- 它会初始化带嵌入能力的专用 `skills` 集合。
- 只有在 LLM 适配器可用时，才启用抽取与演化。
- `QualityGate` 只有在配置了 LLM 适配器时才存在；没有它时，候选项也可能在不经过该门控的情况下被持久化。
- Sandbox TDD 受配置控制，且默认关闭。
- 技能使用事件会被独立存储，并结合 `trace` 结果评估后更新指标与奖励分数，但这类信号噪声较大（缺失或失败的 `trace` 会拉低完成度判断）。

当前技能检索只是通用检索结果中的 `top-k` 合并，而不是单独的检索模式或规划阶段。

## 开放边界

- 系统没有基于质量或使用情况的自动审批或自动提升；仍然依赖显式的人工作业。
- 除谱系元数据外，没有版本回滚或 diff 工具；修订型 `fixed` 技能会创建新的候选版本，但不会在状态变更之外管理多版本共存。
- 明确禁止跨租户共享；可见性范围受租户 `tenant` 以及（对私有技能而言的）`user` 限制。
- 抽取尚未接入执行轨迹，而 `SkillEvaluator` 的奖励信号目前仍是轻量且有噪声的。
- 技能生命周期已经存在，但围绕它的运营治理能力（评审流程、审计轨迹、外部策略钩子）尚未实现。
