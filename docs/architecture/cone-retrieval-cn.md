# 锥形检索（Cone Retrieval）

## 为什么它存在

纯向量相似度可能会漏掉这样一类记录：它们通过共享实体紧密相关，但在嵌入空间中并不相邻。锥形检索在常规检索之上增加了一层轻量级的实体邻域扩展，让共享具体人物、系统或制品的相关记忆能在排序中一并被带出。

## 核心组件

- 写入/更新时的实体提取：有 LLM 可用时，`MemoryOrchestrator` 会通过 `_derive_layers` 和分层派生提示词提取实体，将其归一化为小写后存入 `record["entities"]`（上限 20 个）。如果未配置 LLM，新写入通常不会派生实体，但锥形检索仍可利用集合中已有的实体继续工作。
- `EntityIndex`：按集合维护的内存倒排索引，映射 `entity -> {memory_id}` 与 `memory_id -> {entity}`。它会在启动时通过滚动读取整个集合完成构建，随后在新增、更新、删除时保持同步。更新时的实体同步是尽力而为的，在边缘情况下可能失败或跳过持久化。只有完整构建完成的集合才会被视为可用于锥形评分的 `ready` 状态。
- `ConeScorer`：负责提取查询实体、通过索引扩展候选，并结合 hop 成本与惩罚项计算锥形加分。
- `HierarchicalRetriever` 集成：在常规检索与重排完成后应用锥形扩展与评分，并把加权后的锥形加分叠加到最终分数中。

## 扩展流程

1. 常规检索先产出初始候选列表（向量 + 词法检索，可选重排）。
2. `ConeScorer.extract_query_entities()` 从当前候选中汇总实体，只保留那些在查询文本中以子串形式出现的实体。
3. 扩展阶段：先针对每个查询实体，从 `EntityIndex` 拉取全部相关 memory ID；然后再对分数最高的前 5 个候选，沿着它们的实体继续扩展，但只纳入度数不超过 `ENTITY_DEGREE_CAP` 的实体。
4. 扩展总量受 `MAX_EXPANSION`（20）限制。新增记录会通过 `storage.get()` 取回，并在追加到候选列表、标记 `_expanded=True` 之前，经过租户、用户与项目三级访问控制过滤。

## 评分模型

锥形评分会为每个候选计算一个代价值。这个值由候选自身基于分数换算出的距离，以及通过与其他当前候选共享实体形成的单跳比较共同决定：

- 基础距离为 `1 - raw_score`，并钳制在 [0, 1] 区间内。
- 当候选没有实体且其 `raw_score` 低于 0.9 时，会增加一个直达命中惩罚项 `DIRECT_HIT_PENALTY`。
- 每个共享实体单跳都会增加 `HOP_COST`，若该实体本身也是查询实体，则此成本减半；同时还会叠加相邻候选的距离。高连接度实体会被忽略，除非它们是查询实体。
- 如果找不到任何共享实体路径，则使用 `EDGE_MISS_COST` 作为兜底；但在正常运行流程里，只要候选非空，直达命中路径总会被加入，因此这项成本基本只会在兜底场景下生效。
- 锥形加分的计算方式为 `1 - min(1, cone_cost)`。

随后，`HierarchicalRetriever` 会把 `cone_weight * _cone_bonus` 加到候选分数上，再按 `_final_score` 重新排序。锥形加分是叠加项，不会替代基础检索分数。

## 失败与降级行为

- 当 `CortexConfig.cone_retrieval_enabled` 为 false、`cone_weight <= 0`，或检索器构造时未提供 `ConeScorer` 时，功能会被禁用。
- `EntityIndex.build_for_collection()` 会在启动时异步执行；如果构建尚未完成，索引不会进入 `ready` 状态，此时锥形评分会回退为直接使用基础分数作为加分。扩展仍会利用当下增量可见的实体继续运行。
- 如果实体提取没有得到任何查询实体，扩展仍然可以通过“前 5 个候选的实体扩展”这条路径补充候选。
- 扩展取回或评分过程中发生的错误都会被捕获；检索器会记录日志并返回常规候选列表。

## 与常规搜索的关系

锥形检索不是独立的检索模式。它是在 `HierarchicalRetriever` 内部、常规候选收集和重排之后、结果转换为 `MatchedContext` 之前应用的。无论是否使用嵌入器，只要启用了锥形评分，都会经过这条路径。因此，锥形检索是对现有搜索流程的增强，而不是替代性的检索管线。

## 约束与权衡

- 它能改善由实体关联驱动内容的召回，但效果依赖实体提取的质量和覆盖率。
- 每个集合都需要维护一份内存索引，并在启动时完成一次构建，这会带来额外的延迟和内存开销。
- 扩展与评分都受限于 `MAX_EXPANSION` 和 `ENTITY_DEGREE_CAP`，以避免图扩张失控；代价是部分长尾邻居不会被扩展出来。
- 扩展会执行访问控制过滤，但不会重新应用调用方的元数据过滤条件，因此可能引入严格过滤范围之外的候选。
- `cone_weight` 可配置，但其他惩罚项和成本常量目前在 `ConeScorer` 中是固定值。

## 当前状态

当 `cone_retrieval_enabled` 为 true（默认）且 `cone_weight` 非零（默认 0.1）时，锥形检索处于启用状态。实体会在写入和更新时被派生、归一化并存入向量载荷；`EntityIndex` 在后台构建，并在后续写入/删除中继续保持同步。锥形评分已经接入 `HierarchicalRetriever`，因此这里描述的是当前运行时行为，而不是未来计划。
